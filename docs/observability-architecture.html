Observability Architecture

4-pillar observability with Grafana stack: metrics, logs, traces, and profiling with cross-pillar correlation

Four Pillars Overview

The observability stack covers four complementary signal types, each handled by a dedicated Grafana ecosystem component running in single-binary mode on a shared EC2 instance.

Metrics Mimir

Port: 9009

Memory: 768 MB

Long-term Prometheus-compatible metrics storage. Handles the 4 Golden Signals (latency, traffic, errors, saturation) and custom application metrics.

Logs Loki

Port: 3100

Memory: 384 MB

Log aggregation with label-based indexing. Stores structured JSON logs with trace_id and span_id fields for correlation.

Traces Tempo

Port: 3200

Memory: 384 MB

Distributed trace storage. Captures full request lifecycle from browser through backend with latency waterfall visualization.

Profiling Pyroscope

Port: 4040

Memory: 384 MB

Continuous profiling with flame graphs. Links CPU/memory profiles to specific traces and spans for root cause analysis.

System Architecture

Data flows from application sources through Alloy (unified collector) to backend storage, with Grafana providing visualization and correlation.

Workspace Mapping

Resources are split across base (persistent) and runtime (ephemeral) workspaces to preserve data across infrastructure lifecycle operations.

Resource Workspace Lifecycle Rationale
S3 Bucket (observability data) Base Persistent Survives runtime destroy/recreate cycles
IAM Role + Instance Profile Base Persistent Stable identity for EC2 instance
Security Group (observability) Base Persistent Referenced by ALB and ECS security groups
EC2 Instance (Grafana stack) Runtime Ephemeral Destroyed with just infra down runtime
ALB Target Groups + Rules Runtime Ephemeral Path routing for /grafana/* and /collect/*
ECS Task Env Vars Runtime Ephemeral OTel endpoint configuration

Component Details

Each Grafana stack component runs in single-binary mode within Docker Compose on a shared t3.medium EC2 instance (4 GB RAM, ~2.7 GB allocated to containers).

Visualization Grafana

Memory: 384 MB

Port: 3001

Mode: Standalone

Unified dashboard and alerting UI. Reads from all 4 backend datasources. Serves from /grafana/ subpath via ALB.

Metrics Mimir

Memory: 768 MB

Port: 9009

Mode: Single-binary

Prometheus-compatible long-term metrics storage. Handles exemplars for trace correlation. S3 block storage.

Logs Loki

Memory: 384 MB

Port: 3100

Mode: Single-binary

Label-based log aggregation. TSDB index with S3 chunk storage. Structured JSON with trace_id/span_id fields.

Traces Tempo

Memory: 384 MB

Port: 3200

Mode: Single-binary

Distributed trace backend. Receives OTLP from Alloy, generates span metrics, enables trace-to-logs/profiles navigation.

Profiling Pyroscope

Memory: 384 MB

Port: 4040

Mode: Single-binary

Continuous profiling with flame graphs. Push-based SDK integration. Links CPU/memory profiles to traces via span labels.

Collector Alloy

Memory: 384 MB

Ports: 4317 (gRPC), 4318 (HTTP), 12347 (Faro)

Mode: Single-binary

Unified telemetry collector. Receives OTLP from backend, Faro from frontend, and forwards to all storage backends.

Cross-Pillar Correlation Paths

The primary value of 4-pillar observability is navigating between signal types to find root causes. These correlation paths enable seamless investigation.

Path From To Mechanism
Metric Exemplar → Trace Mimir Tempo Exemplar config links metric data points to trace IDs
Trace → Logs Tempo Loki traceToLogs config filters Loki by trace_id label
Trace → Profile Tempo Pyroscope traceToProfiles config links to Pyroscope via span labels
Alert → Dashboard → Trace → Root Cause Grafana Alerting All Pillars Full investigation workflow: alert fires, click through to dashboard, exemplar to trace, trace to logs and profiles

Network & Security

The observability EC2 instance runs in a private subnet with no public IP. All external access routes through the ALB. Management access uses SSM Session Manager.

Port Protocol Source Purpose
3001 HTTP ALB Security Group Grafana UI via /grafana/* path
12347 HTTP ALB Security Group Faro telemetry collection via /collect/* path
4317 gRPC ECS Security Group OTLP telemetry from backend (gRPC)
4318 HTTP ECS Security Group OTLP telemetry from backend (HTTP)
4040 HTTP ECS Security Group Pyroscope push endpoint for profiling data

Security Posture

  • Private subnet placement: EC2 has no public IP; accessible only via ALB or SSM
  • ALB as sole external access: Path-based routing restricts which services are exposed
  • S3 bucket policy: Blocked public access, instance profile grants write permission
  • Grafana anonymous access: Limited to viewer role (read-only dashboards)
  • SSM Session Manager: Management access without SSH keys or bastion hosts

Cost Analysis

All observability resources are gated behind the enable_observability feature flag. When disabled, the additional cost is zero.

Resource Monthly Cost Conditional
EC2 t3.medium ~$30 enable_observability
S3 storage <$1 (dev) enable_observability
ALB rules $0 (reuses existing) enable_observability
Total enabled ~$31
Total disabled $0

The EC2 instance follows the same ephemeral lifecycle as other runtime resources—destroyed during off-hours and recreated on demand. S3 data in the base workspace persists across these cycles.
