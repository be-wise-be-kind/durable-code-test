Terraform Architecture

Understanding the 3-workspace design pattern for AWS infrastructure management

Overview

The infrastructure is organized into three Terraform workspaces, each managing a distinct layer of AWS resources. This separation enables:

  • Cost optimization through selective teardown of expensive resources
  • Risk isolation by keeping permanent resources separate from ephemeral ones
  • Clear ownership of resources and their lifecycle

Workspace Details

Bootstrap Workspace

Purpose: Foundation infrastructure that enables all other deployments.

Lifecycle: Created once, never destroyed. Manual changes only.

State Path: bootstrap/{env}/terraform.tfstate

Base Workspace

Purpose: Core networking and supporting services that persist across deployments.

Lifecycle: Created once per environment, rarely changed.

State Path: base/{env}/terraform.tfstate

Runtime Workspace

Purpose: Application workloads and expensive resources that can be torn down.

Lifecycle: Destroyed nightly, recreated on demand.

State Path: runtime/{env}/terraform.tfstate

State File Organization

All Terraform state is stored in a centralized S3 bucket with the following structure:

Workspace State File Path Description
Bootstrap bootstrap/dev/terraform.tfstate OIDC, IAM roles, state bucket itself
Base base/dev/terraform.tfstate VPC, subnets, ECR, Route53, ACM
Runtime runtime/dev/terraform.tfstate NAT, ALB, ECS, WAF, logs
Backend Configuration Example ▼ 
# infra/terraform/backend-config/runtime-dev.hcl

bucket               = "durable-code-terraform-state"
key                  = "runtime/dev/terraform.tfstate"
region               = "us-west-2"
encrypt              = true
dynamodb_table       = "durable-code-terraform-locks"
workspace_key_prefix = ""   # Disables env:/ prefixing

Up/Down Workflow

The infrastructure supports a cost-saving pattern where expensive resources are destroyed outside business hours.

Down Workflow (Teardown)

Scheduled daily at 8 AM UTC (midnight PST) via down.yml, or triggered manually.

1

Authenticate

GitHub Actions authenticates via OIDC to assume the Terraform deploy role

2

Initialize

Run terraform init -backend-config=runtime-dev.hcl

3

Destroy Runtime

Run terraform destroy on runtime workspace only

4

Verify

Confirm NAT Gateway, ALB, and ECS resources are removed

Up Workflow (Startup)

Scheduled daily at 3 AM UTC (8 PM MST) via up.yml, or triggered manually.

1

Authenticate

GitHub Actions authenticates via OIDC to assume the Terraform deploy role

2

Initialize

Run terraform init -backend-config=runtime-dev.hcl

3

Apply Runtime

Run terraform apply to create NAT, ALB, ECS, and related resources

4

Wait for Health

ALB becomes available (~5 min), ECS services start, health checks pass

Cross-Workspace Dependencies

The runtime workspace reads outputs from the base workspace using Terraform remote state:

Remote State Reference Code ▼ 
# infra/terraform/workspaces/runtime/data.tf

data "terraform_remote_state" "base" {
  backend = "s3"

  config = {
    bucket = "durable-code-terraform-state"
    key    = "base/${local.environment}/terraform.tfstate"
    region = "us-west-2"
  }
}

# Usage example
data "aws_vpc" "main" {
  id = data.terraform_remote_state.base.outputs.vpc_id
}

Cost Impact

The up/down pattern saves approximately $63+/month by destroying:

Resource Monthly Cost Destroyed Nightly?
NAT Gateway ~$45/month Yes (runtime)
Application Load Balancer ~$18/month Yes (runtime)
ECS Fargate Tasks Variable Yes (runtime)
VPC, Subnets, Route53 ~$0 No (base)
ECR Repositories ~$0 No (base)
